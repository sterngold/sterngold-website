I Automated My Morning Context Load. Here's What I Learned.

Vlad Sterngold · February 2026 · 4 min read

For the first few weeks of using Claude Code seriously, I noticed I was losing 10–15 minutes at the start of every session doing the same things.

Open the action items file. Check git. Scan email. Try to remember where I left off yesterday. Mentally reconstruct the priority stack. By the time I was actually thinking, a significant chunk of the morning was already gone.

The irony wasn't lost on me: I was using an AI coding assistant, and I was spending my first fifteen minutes being a human file system.

So I built a skill to handle it.

What a “skill” is

Claude Code has a system for custom instructions called skills — markdown files stored in ~/.claude/skills/ that the model loads and executes when you type a trigger phrase. They're natural-language programs. No code, no deployment, no dependencies. Just structured instructions that tell the model what to do and in what order.

I already had a session-wrap skill for closing sessions — it commits code, updates my context files, runs a short retrospective. What was missing was the mirror: something to open a session.

I called it session-start.

What it does

When I type /start, Claude runs five phases silently and produces a single Daily Brief. The whole thing takes about 20 seconds.

Phase 1 — Context sync. Reads my action items, goals, and work context files. Extracts open items and deadlines. Identifies anything time-sensitive. Produces no output — this feeds the brief.

Phase 2 — Workspace check. Runs git status. Lists any uncommitted files. Never auto-commits — just flags. If the workspace is clean, it notes that and moves on.

Phase 3 — Calendar scan. Fetches today's events from my iCloud calendar via a public iCal URL — a curl request and a few lines of Python to parse it. No calendar app integration needed, no API credentials. Just a URL and a date filter.

Phase 4 — Email triage. Hits the Gmail API, pulls 15 message metadata records first (subject + sender only), filters out newsletters and automated mail, then reads full content only for human-sent messages. Maximum 5 full reads. Flags anything requiring action.

Phase 5 — Daily Brief. One structured markdown output: workspace state, calendar, email flags, top 3 priorities ranked by deadline, and a concrete recommendation for what to do first.

The design decisions worth knowing

Metadata before content. The email phase fetches subjects and senders for all 15 messages before reading anything in full. This means the model can filter out newsletters without loading their bodies into context. Saves tokens, keeps the brief tight. It's a small thing, but it matters when you're doing this every morning.

Fallbacks with no drama. I wrote the skill to degrade gracefully. If the calendar URL fails, it says so briefly and moves on. If there's no unread email, it says “inbox clear” — one line, no commentary. The brief happens regardless.

Instruction documents, not code. The skill file is pure markdown: phase descriptions, decision logic, edge case handling. There's no interpreter, no runtime, no deployment step. Claude reads it and executes it. The tradeoff is you can't loop or retry — but for a morning brief, you don't need to.

The priority ranking

The brief ranks the top three open action items by a cascade I made explicit in the skill instructions:

  1. Items with hard deadlines — soonest first
  2. Items marked urgent
  3. Active external dependencies — things waiting on other people, legal timelines, financial decisions
  4. Items that unlock other items
  5. Domain priority — roughly: financial and legal before business, business before creative

The ranking sounds obvious when you write it out. That's the point. Encoding it means the model applies it consistently rather than defaulting to whatever feels most salient in the moment — which is usually the most recently mentioned thing, not the most important one.

What changed

The session starts with a brief now instead of a blank slate. I know what's on my calendar before I open the calendar. I know if anything urgent arrived before I open email. I know the git state before I type a command.

The real value isn't the brief itself. It's that the brief forces the context loading that I'd otherwise do scattered across fifteen minutes of tab-switching. It's the same information — just structured, front-loaded, and not dependent on my memory being reliable at 8am.

The whole skill file is about 100 lines of markdown. It took one session to build and debug.

I've been running it every morning since.
